# Impeccable Design Context

This file captures the product context that any UI redesign must respect.
It is consumed by the `/impeccable` design skill at the start of every
design session.

## Design Context

### Users

A small internal team of HR / operations admins (1 primary + a handful of
secondary admins) running a driver-referral program for a transportation
company. They open the Mini App inside Telegram on a phone, mid-shift,
to do one of a few short jobs:

- **One-time announcement** — pick a referrer, hit broadcast, watch the
  spotlight go out to every approved chat.
- **Weekly / scheduled announcement** — set the days + time once and
  trust it to fire.
- Add or clean up driver referrals tied to a referrer, adjust CPM, export
  the spreadsheet for payroll.
- Primary admin only: manage approved admins and approved chats.

Sessions are short (under a minute), fingers-on-glass, often one-handed.
Desktop is secondary but supported (the same admin reviewing on a laptop).

### Brand Personality

**Celebratory · warm · direct.**

The product's emotional core is *recognition* — referrals are how drivers
get more money and more CPM, and announcements are public victory laps.
The UI should feel like writing a thank-you note, not filing a TPS
report. Direct because admins are busy and trust is earned by getting
out of their way.

### Aesthetic Direction

- **Reference**: Notion. Specifically the calm, generous, type-led
  feeling — soft surfaces, real type hierarchy, no chrome for the sake
  of chrome, controls that look like part of the document.
- **Anti-reference**: enterprise SaaS dashboards (cyan-on-dark,
  purple-to-blue gradients, dense KPI tiles, candy-colored icons in
  rounded squares above every heading). No glassmorphism. No
  border-left accent stripes. No gradient text.
- **Theme**: light by default with a tuned dark variant — driven by
  Telegram's `colorScheme` so it matches the surrounding chat. Both
  variants must feel intentional, not auto-generated from each other.
- **Color**: warm low-chroma neutrals tinted toward the brand hue, with
  a single accent reserved for the celebratory beat (the broadcast CTA,
  the leader's badge, the success state). Accent is a deliberately
  off-axis warm tone — not SaaS blue, not bro-green. Picked at craft
  time within an OKLCH palette and tinted into neutrals.
- **Type**: paper-document feel. A serif or humanist display face for
  headings paired with a refined humanist sans for UI text. Reflex
  fonts (Fraunces, Inter, DM Sans, Plus Jakarta, Instrument Serif, etc.)
  are off the table.
- **Motion**: minimal and only at meaningful state changes — modal
  open, item enter/leave, "sent!" feedback. No decorative motion. Honor
  `prefers-reduced-motion`.

### Most-used flow

Sending an announcement (one-time or scheduled) is the dominant action
and must be reachable in **one tap from app open**. Everything else can
sit a level deeper.

### Design Principles

1. **Celebrate the act, not the dashboard.** The interface should make
   sending a spotlight feel like a small ceremony — clear stakes, clean
   confirmation. Decoration that doesn't reinforce that beat is noise.
2. **One thumb, mid-shift.** Mobile-first, single-column, generous tap
   targets, the primary action always within thumb reach. Desktop is a
   centered narrow column with breathing room — not a wider grid.
3. **Type does the heavy lifting.** Hierarchy comes from weight, size,
   and color — not boxes, dividers, or icon chrome. No card-in-card.
4. **One accent, used rarely.** The celebratory color earns its keep by
   being scarce. Everything else is warm neutral.
5. **Trust the platform.** Use Telegram's `showPopup` /
   `showConfirm` / `showAlert` and `MainButton` instead of custom modal
   chrome wherever it's a clean fit. Don't reinvent native affordances
   the user already knows.
6. **Safe by default.** No HTML interpolation of user-supplied text.
   Render data with `textContent` / `createElement`, never
   `innerHTML +=`. (Closes the XSS sinks present in the current
   implementation.)

### Out of scope for the redesign

- Changing the Python API surface (`/api/*` endpoints stay the same).
- Adding new features. The redesign re-skins, restructures, and
  hardens the existing functionality only.
- Build tooling. Output stays vanilla HTML/CSS/JS served by aiohttp.
